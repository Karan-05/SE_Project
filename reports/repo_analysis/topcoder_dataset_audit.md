# Topcoder Dataset Audit

## 1. Provenance of the 22,023 rows
- **API collectors** – `init.py` + `setUp` (`init.py:20`, `setUp.py:16`) wrap the v5 `/challenges` REST API with pagination, status/track filters, and retry logic (`fetch_functions.py:27`). All API downloads are normalised via `process.format_challenge` and written to `challenge_data/challengeData_<start>_<end>/pageN.json`.
- **Historical Excel** – `legacy_excel_loader.py:135` converts `old_Challenges.xlsx` into JSON using the same schema as the API, preserving IDs, timestamps, technology tags, and prize pools. On 2026‑03‑07 we regenerated the file with `python legacy_excel_loader.py old_Challenges.xlsx --output-dir challenge_data --window-name legacy_xlsx --output-file page1.json`, yielding 34,685 rows and 21,819 unique `challengeId`s.
- **Member/context exports** – `snapshots/Challenge_Member_Mapping.csv`, `snapshots/Challenges.csv`, and `snapshots/Members.csv` are MySQL exports (generated via `dbConnect.excel_uploader`, `dbConnect.py:205`) that capture whichever registrant, submission, and profile rows were ingested when the exporter last ran.

## 2. Reproducing the “22,023 challenges” claim
1. **Regenerate legacy JSON** (step above) and ensure the API windows live under `challenge_data/` (currently Jan 2023 and Jun–Oct 2025).

## 3. Authoritative artefacts
- **Raw challenge payloads** – JSON under `challenge_data/` (API windows plus the regenerated legacy export) are the canonical source for challenge metadata.
- **Relational truth** – The MySQL schema defined in `schema_registry.py:12` (tables `Challenges`, `Challenge_Member_Mapping`, `Members`) is authoritative whenever a DB is available. CSV/Excel snapshots in `snapshots/` are the frozen state of those tables when `dbConnect.excel_uploader` last ran.
- **Analysis outputs** – `analysis/output/*.csv` and `analysis/output/report.md` are deterministic derivations of the raw JSON + member mapping, and they retain a `source_file` column so every aggregate can be traced back to its raw page.
- **Research tables** – `data/raw/{tasks,workers,interactions}.csv` (generated by `scripts/export_real_tasks.py`) and `data/processed/*.parquet` (from `src/data/preprocess.py`) are the starting point for downstream modelling. On 2026‑03‑08 we verified that both artefacts cover the full 22,023-challenge corpus (`data/raw` CSV row counts match `data/processed/metadata.json`’s `num_tasks=22023`, `num_workers=402280`, `num_interactions=425704`).

## 4. Deduplication & hygiene logic
- **Collector-level** – `analysis/report.load_local_challenges` (`analysis/report.py:389`) keeps a dict keyed by `challengeId`, so if a legacy and API window both contain the same challenge the later file wins but duplicates cannot inflate counts. The same keyed dict exists in `scripts/export_real_tasks._iter_challenge_payloads` to prevent double-emission.
- **Database-level** – MySQL tables declare unique keys on `challengeId`/`memberHandle` (`migrations/001_initial_schema.sql`), and `dbConnect.upload_data` issues `INSERT ... ON DUPLICATE KEY UPDATE`. `Uploader.check_unique_members` respects TTLs before fetching members again, preventing stale duplicates (`uploader.py:126`).
- **Registrants/submissions** – `load_member_mapping` (`analysis/report.py:469`) only counts rows when the mapping CSV has at least the six expected columns; anything malformed is skipped, which is why registrant coverage today is limited to the 3,218 rows present in the snapshot.

## 5. Gaps, risks, and missing credentials
- **Bearer token absence** – `fetch_challenge_submissions` short-circuits without `TOPCODER_BEARER_TOKEN`, so `analysis/output/report.md` currently shows “Members with submissions: 0” despite 20,225 challenges reporting submissions counts. Submission identities (and thus override win rates) remain unknown until a token is supplied.
- **Registrants coverage** – `snapshots/Challenge_Member_Mapping.csv` contains only a few thousand relationships, so aggregated registrant counts (Unique active members = 576) reflect the snapshot, not the entire 22k corpus.
- **Database credentials** – No `.env` or `TOPCODER_DB_*` secrets are included. Rebuilding the MySQL schema requires an accessible server plus env vars (`config.py:34`).
- **Processed tables maintenance** – The refreshed `data/raw/` export and `data/processed/metadata.json` are aligned today, but any future edits to `challenge_data/` require re-running both `scripts/export_real_tasks.py` and `python -m src.data.preprocess` to keep downstream tables consistent.
- **Legacy weighting** – 99% of rows originate from the legacy Excel dump; fresh API windows cover only 204 modern challenges. Any statistics should call out that imbalance and, if needed, re-fetch missing years via `automation.Automation` to avoid over-reliance on the legacy snapshot.
